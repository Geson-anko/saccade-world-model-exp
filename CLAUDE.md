# CLAUDE.md

このリポジトリで作業するエージェント／開発者向けのガイダンス。プロジェクトの全体像と動機は [README.md](README.md) を参照。本ファイルは **決定事項とその理由（再議論しないための記録）** と、守るべき不変条件を中心に記す。

## 一行で

サッカード（注視点 `p` ＋ 拡大率 `z`）で画像の部分観測を取り、その系列から画像全体の内部表現を組み上げる、**文脈内・自己回帰の潜在世界モデル**の実験。

## 決定事項と理由（変えるときは要相談）

これらは設計の背骨。安易に崩さない。

1. **汎化（in-context）であり、1 枚のメモライズではない。** 多数の画像で学習し、未知の画像でも数枚のグリンプスから未観測グリンプスを文脈内予測する。各画像が独立した 1 エピソード。
2. **自己回帰。** 履歴 `(a_{≤t}, e_{≤t})` ＋ 次行動 `a_{t+1}` → `ê_{t+1}`。順序に意味を持たせる。
3. **予測は潜在空間で行う。生ピクセルは直接予測しない。** 画素レベルのズレという本質的でない誤差が表現学習を汚すため。
4. **stop-grad なし・完全同時学習。** ターゲットはオンラインの `E(o_{t+1})`。崩壊は **SIGReg**（埋め込み分布への正則化）で防ぐ。teacher-student / EMA / stop-grad に頼らない（LeJEPA 流）。
5. **Encoder は観測 `o` のみを見る。** `a` は系列モデル側で注入する。
6. **デコーダは可視化・評価専用の detached read-out。** Encoder / 系列モデルへ勾配を返さない。ピクセル損失はデコーダ内部のみ。デコーダは「内部表現に画像情報が含まれるか」を測る道具であって、世界モデルの目的関数ではない。
7. **行動はランダムサンプリング。** 能動的な「次にどこを見るか」ポリシーは当面扱わない。
8. **境界はパディングで一様化。** `image (size,size)` を各辺 `size/2` 黒パディングして `(2·size, 2·size)` にし、任意の `(p,z)` を場合分けなしにサンプルする。

## 中心的な問い（この実験のゴール）

系列モデルの内部状態 `b_t`（belief）**単体から、ソース画像をデコードできるか**。観測列を通じて画像全体を完備した内部表現として保持できるか、を検証する。評価は層 1（次グリンプスの潜在予測）と層 2（`b_t` からの全体画像復元）の 2 層。

## 座標・記号の規約

- `p = (x, y)`, `x,y ∈ [-1,1]`：画像中心からの注視点。`size/2 ↔ 1`。
- `z ∈ [0,1]`：切り取り辺の長さ。`size ↔ 1`、`z = 0` で 1 ピクセル。切り取り辺 ≈ `z · size`。
- `a = (p, z)` 行動 / `o` 観測 `(s', s')` / `e = E(o)` 埋め込み / `b_t` belief。
- 観測は常に固定サイズ `s' ≤ 128`。予測は `t ≥ 2`（`t=1` は情報ゼロ）。

## モデル構成（現時点の方針）

- **系列モデルは差し替え可能** にする。主軸 **minGRU**（固定長状態 = belief、圧縮の純度が高い）、比較対象 **Transformer**（belief 読み出し機構は後で検討）。
- `b_t` の取り出し方は未確定（末尾隠れ状態 / 読み出しトークン / pooling など）。実装時に比較して決める。

## 開発原則（最重要）

書き方より先に**働き方**。コードの良し悪しはこの 4 つで判断する。

1. **実装前に考える。** 仮定を勝手に置かない。混乱を隠さない。トレードオフを表に出す。不確実な点は質問し、複数の解釈が可能なら全て提示してから着手する。
2. **シンプルさを優先。** 問題を解く最小限のコード。投機的な実装はしない。頼まれていない機能追加や過度な抽象化を避ける。200 行を 50 行で実装できるなら書き直す。
3. **外科的な変更。** 触る必要があるものだけ触る。自分が散らかしたものだけ片付ける。既存スタイルに合わせ、自分の変更で生じた orphan のみ削除する。無関係な「ついで改善」をしない。
4. **ゴール駆動の実行。** 成功条件（＝検証できる条件、多くはテスト合格）を最初に定義し、そこへ向けて短い計画で段階的に進める。検証できるまでループし、確認してから完了とする。

## コーディング規約

- **private モジュール規約。** テストの有無で `_` prefix を決める。テストを書かない実装ファイル → `_prefix`、テストを書く公開対象 → prefix なし。モジュールの外部公開面は `__all__` で明示管理する。
- **カプセル化。** 内部実装・属性は原則 private（`_`）。外部参照が必要なものだけ public にする。
- 既存コードに**馴染む**書き方をする（命名・コメント密度・イディオムを周囲に合わせる）。

## Git 運用

- **`main` が主軸。** 作業はブランチを切る: `<種別>/<日付>/<内容>`（例 `feat/2026-06-27/saccade`）。
- **コミットメッセージ:** `<種別>(<スコープ>): <内容>`（例 `feat(saccade): パディング付き切り取りを実装`）。
- 検証必須。**テストが落ちている状態でコミットしない。**

## 技術スタック / 開発

- Python `>= 3.13`（`.python-version` は 3.13）、パッケージ管理は **uv**。**配布物ではない**ため `tool.uv.package = false`（ビルドしない）。
- 開発は **devcontainer（docker-compose・NVIDIA GPU）** 上で行う（VS Code "Reopen in Container"）。
- モジュール系ソースは **フラットな `exp/`**（src-layout にしない）。学習・評価のエントリーポイントは `scripts/` かリポジトリルート。
- コマンドは **just** に集約: `just setup` / `format` / `lint` / `type` / `test` / `run`。依存追加は `uv add <pkg>`、任意実行は `uv run <cmd>`。
- 品質ゲートは **ローカルの `just run`**（format → test → type）。**GitHub CI は持たない**（配布物でないため）。lint/format は pre-commit に集約（ruff ほか）。
- ランタイム依存は **torch / torchvision**（GPU 学習）＋ numpy / attrs。依存追加は `uv add <pkg>`。開発フローの詳細は `.claude/skills/dev-workflow` を参照。
